# Contributing

## Documentation rules

- Keep statements concrete and tied to code or observed schema usage.
- Mark uncertainty explicitly with `confirmed`, `inferred`, or `uncertain`.
- Use exact SQL object names and file paths.
- Do not paste real secrets.

## Config documentation standard

For each runtime-affecting config field, document:

- name
- table and column
- type or observed type
- allowed values
- default or observed default
- null behaviour
- where it is read
- functional effect
- side effects
- dependencies
- live-change safety
- verification notes
- risk and confidence

## Generated pages

- `docs/configuration/reference.md`
- `docs/configuration/flags/*.md`

These are generated by `docs/scripts/generate_config_docs.ps1`. A Python version also exists for environments that already have Python installed.

After editing runtime config behaviour, run:

```powershell
cd docs
powershell -File scripts/generate_config_docs.ps1
mkdocs build --strict
```

## Mermaid diagrams

- Use fenced blocks with the `mermaid` language for flowcharts and sequence diagrams.
- The docs site renders Mermaid at build and runtime through MkDocs Material plus `pymdownx.superfences`.
- If a diagram shows as raw text, rebuild with `mkdocs build --strict` and inspect the generated HTML for `<div class="mermaid">` instead of a highlighted code block.

## Code references in docs

- Prefer exact script names such as `Sync-ConfiguredSqlTable.ps1`.
- Include line numbers when they materially help troubleshooting.
- Distinguish between direct code confirmation and inference from naming.
